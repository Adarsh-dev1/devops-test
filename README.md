# DevOps Take-Home Assignment

## Overview

This project is a minimal Node.js Express application that has been:

* Containerized using Docker
* Configured to run with Docker Compose
* Automated with a GitHub Actions CI pipeline
* Equipped with basic request logging using Express middleware (Morgan)
* Tested with Jest and Supertest

---

---

# Running Locally (Without Docker)

## Install dependencies

```bash
npm install
```

## Start the application

```bash
npm start
```

The application will be available at:

* http://localhost:3000/
* http://localhost:3000/health

---

# Running with Docker

## Build the image

```bash
docker build -t devops-test .
```

## Run the container

```bash
docker run -p 3000:3000 devops-test
```

---

# Running with Docker Compose

If the `docker-compose.yml` file is configured with a build context:

```yaml
build: .
```

run:

```bash
docker compose up --build
```

If the build context is omitted, ensure the Docker image has already been built before running:

```bash
docker build -t devops-test .
docker compose up
```

To stop the application:

```bash
docker compose down
```

# Running Tests

Execute the test suite with:

```bash
npm test
```

The test verifies that the `/health` endpoint responds with HTTP status code **200**.

---

# CI Pipeline

The GitHub Actions workflow runs automatically on every push.

The pipeline performs the following steps:

1. Checks out the repository.
2. Sets up the required Node.js environment.
3. Installs project dependencies using `npm ci`.
4. Executes the test suite.
5. Builds the Docker image to verify that the application can be successfully containerized.

The pipeline intentionally executes tests **before** building the Docker image so that image creation is skipped if the application fails its tests.

# Request Logging

The application uses **Morgan** as Express middleware to log incoming HTTP requests.

Each log entry contains:

* HTTP Method
* Request Path
* HTTP Status Code
* Response Time

Example:

```text
GET /health 200 1.84 ms
```

# Production Monitoring

Assuming the application is deployed to production as a fully completed application:

In a production environment, I would enable centralized application logging using a Prometheus sidecar or DaemonSet (depending on the deployment architecture) and create dashboards to monitor key metrics such as request rates, error rates, and response latency. The application's request logs generated by the middleware would be collected, and HTTP error status code ranges (4xx and 5xx) would be monitored to identify abnormal behavior.

I would also implement application performance monitoring (APM) using OpenTelemetry to collect traces and performance metrics. Alerting and notification systems would be configured to notify the operations and development teams when predefined thresholds are exceeded, such as high error rates, increased latency, or request timeouts.

Additionally, assuming the completed application includes application-level logging, I would ensure that sensitive information is properly masked before logs are collected. If the error rate or other critical metrics exceed defined thresholds, automated alerts would be sent to the development team for investigation and remediation.

# Assumptions and Trade-offs

* The application is minimal, so a separate build phase is not added.
* The CI pipeline executes the test step before building the Docker image, preventing unnecessary image builds if the tests fail.
* The Docker image is built to verify successful containerization but is not pushed to a container registry, since it was not mentioned.
* Morgan was selected for request logging because it is lightweight, widely used, and integrates seamlessly as Express middleware.
